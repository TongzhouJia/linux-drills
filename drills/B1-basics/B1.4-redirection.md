# KP B1.4 — Redirection (advanced) / 重定向进阶
**`>` `>>` · `2>` · `2>&1` 顺序陷阱 · `&>` · `/dev/null` · here-doc · here-string · `tee` · `exec`**

**10 tasks · ⭐⭐ · RHEL 10 · server1 as root · run each in a bash shell and match the stated result**

> 三条独立的"水管"：**0=stdin、1=stdout、2=stderr**。运维脚本能不能把日志、错误分开收好，全看这一节。重点体会第 4 题的**顺序陷阱**——面试和线上都常踩。

---

**1.** Show `>` (truncate/create) vs `>>` (append): put `line1` into `out.txt` with `>`, append `line2` with `>>`. Then re-run an `echo ... > out.txt` and show the file was truncated back to a single line.
*Acceptance:* after the first two, `out.txt` has `line1` then `line2`; after the re-run with `>`, only the new single line remains.

**2.** Send stdout and stderr to **separate** files: run `ls /etc /nope` so the good listing goes to `good.txt` and the error goes to `err.txt`.
*Acceptance:* `good.txt` holds the `/etc` listing; `err.txt` holds `ls: cannot access '/nope': No such file or directory`.

**3.** Send **both** streams into **one** file: capture stdout+stderr of `ls /etc /nope` into `all.txt`.
*Acceptance:* uses `> all.txt 2>&1`; `all.txt` contains both the listing and the error line.

**4.** The **ORDER trap** — demonstrate and explain:
- `ls /nope 2>&1 > t.txt` → the error does **not** end up in the file.
- `ls /nope > t.txt 2>&1` → the error **does** end up in the file.
*Acceptance:* you can show version A leaves the error on the terminal while version B captures it; and you can explain *why* (redirections are applied left→right; `2>&1` duplicates whatever stdout points at **at that moment**).

**5.** Discard **all** output (stdout+stderr) of `ls /etc /nope` using `&>/dev/null`, but keep the exit status usable (print `$?` afterward).
*Acceptance:* nothing is printed; `$?` still reflects the failure (non-zero).

**6.** Suppress **only** stderr: run `find /etc -name '*.conf'` with `2>/dev/null` so "Permission denied" noise is hidden but real matches still print.
*Acceptance:* matching paths print; no permission-denied lines appear.

**7.** here-doc (**expanding**): feed a multi-line block to `cat` with `<<EOF` where `$USER` and `$(date +%F)` are expanded.
*Acceptance:* the printed block shows your username and today's date, not the literal `$USER`/`$(...)`.

**8.** here-doc (**literal**): same block but with `<<'EOF'` (quote the delimiter) so `$USER` and `$(date +%F)` stay literal.
*Acceptance:* the block prints the text `$USER` and `$(date +%F)` verbatim.

**9.** here-string: feed a string straight into a command's stdin with `<<<`. Count the words in `one two three`.
*Acceptance:* `wc -w <<< "one two three"` → `3`.

**10.** `tee` and `exec`:
- Run `echo hello | tee tlog.txt` so `hello` appears **on screen and** in the file; then append `world` with `tee -a`.
- In a script snippet, put `exec > run.log 2>&1` at the top and show that everything printed afterward (including errors) lands in `run.log`.
*Acceptance:* `tee` prints to screen and writes the file; `tee -a` appends; after `exec > run.log 2>&1`, later stdout **and** stderr go to `run.log`.

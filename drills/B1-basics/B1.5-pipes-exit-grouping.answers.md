# KP B1.5 答案 — Pipes · Exit Status · Command Grouping

> 命令都在真实 bash 里跑过、输出核对无误。核心：**0=成功、非0=失败**；`&&` 走成功路、`||` 走失败路；管道的 `$?` 默认只看**最后一段**。

---

**1.** 管道：一段的输出喂给下一段的输入。
```bash
grep '/bin/bash$' /etc/passwd | wc -l     # → 例如 11（各机不同）
```
*为什么：* `|` 把左边命令的 **stdout** 接到右边命令的 **stdin**。`grep` 挑出用 `/bin/bash` 的行，`wc -l` 数行数。管道里每一段都是独立进程。

**2.** `$?` = 上一条命令的退出码。
```bash
ls /etc >/dev/null; echo $?      # → 0    成功
ls /nope 2>/dev/null; echo $?    # → 2    失败（非 0）
```
*为什么：* 约定 **0=成功**，1–255=各种失败。这是所有条件判断的基石。注意 `$?` 会被**下一条命令刷新**，要用就立刻取。

**3.** `&&`：前一条成功才跑后一条。
```bash
mkdir -p /tmp/x && echo created      # → created
```
*为什么：* `A && B` 中，只有 `A` 退出码为 0 才执行 `B`。适合"建目录成功→再往里写"这种依赖链。

**4.** `||`：前一条失败才跑后一条。
```bash
ls /nope 2>/dev/null || echo "not found"     # → not found
```
*为什么：* `A || B` 中，`A` 失败（非 0）才执行 `B`。常用于兜底/报警：`command || { echo fail; exit 1; }`。

**5.** `;` 无条件顺序 vs `&&` 有条件。
```bash
false; echo A         # → A     （; 不看成败，照跑）
false && echo A       # →       （false 失败，&& 后面被跳过）
```
*为什么：* `;` 只是"一条接一条"，`&&`/`||` 才看退出码。别把 `;` 当 `&&` 用，否则前一步失败了后一步还硬跑。

**6.** ⭐ 管道退出码默认 = 最后一段（易漏的失败）。
```bash
false | true; echo "exit=$?  PIPESTATUS=[${PIPESTATUS[*]}]"
# → exit=0  PIPESTATUS=[1 0]
```
*为什么：* 默认 `$?` 只反映**管道最后一段**（`true`→0），中间的 `false` 失败被**掩盖**。`PIPESTATUS` 数组存**每一段**的退出码（`false`=1、`true`=0）。
> ⚠️ 二次坑：`$?` 和 `PIPESTATUS` 只描述**最近一条**管道，会被下一条命令覆盖——必须在管道**紧接的第一条命令里**读取。实测：紧跟读 `PIPESTATUS=[1 0]`，再读一次就变成 `[0]`（反映了中间那条 echo）。

**7.** `pipefail`：让管道"有一段失败就算失败"。
```bash
set -o pipefail
false | true; echo "exit=$?"      # → exit=1   （报出了 false 的失败）
set +o pipefail
```
*为什么：* 开 `pipefail` 后，管道退出码取"最后一个非 0"。运维脚本处理 `cmd | grep | awk` 这种链条时，**必须开 pipefail**（配合 `set -e`），否则前面的命令挂了、脚本还以为成功。这是健壮脚本三件套 `set -euo pipefail` 的一员（B3.1 细讲）。

**8.** `( )` 子shell：里面的改动不外泄。
```bash
x=1; ( x=99; cd /tmp ); echo "$x $PWD"
# → 1 <原目录>       （x 没变、目录没变）
```
*为什么：* `( )` 开一个**子shell**执行，变量赋值、`cd`、`set` 等都只在子进程里生效，退出即丢弃。适合"临时进某目录干点事、不影响主流程"：`( cd /tmp && tar xf a.tar )`。

**9.** `{ }` 命令组：在**当前** shell 执行。
```bash
{ echo a; echo b; } > both.txt      # 两条 echo 的输出一起进文件
wc -l < both.txt                    # → 2
```
*为什么：* `{ }` 把多条命令当一个整体（共用一次重定向），但**不开子shell**——里面的变量赋值、`cd` 会保留。语法要求：`{` 后有空格、`}` 前一条命令要有 `;`（或换行）。
> 一句话记：**要隔离用 `()`，要影响当前 shell 用 `{}`。**

**10.** `A && B || C` 的陷阱（别当 if/else 用）。
```bash
true  && echo yes || echo no     # → yes
false && echo yes || echo no     # → no
```
看着像 if/else，但**有坑**：如果 `A` 成功、可 `B`（`echo yes`）**自己失败**了，`|| C` 会**照样触发**，于是 yes 和 no 都可能出来。
```bash
# 危险示例：B 会失败时
true && (exit 3) || echo "C ran!"   # → C ran!   （A 成功，B 失败，C 却跑了）
```
*正解：* 逻辑重要时用真正的 `if`：
```bash
if command; then
  echo yes
else
  echo no
fi
```
`&& ||` 只适合"两边都绝不会失败"的极简场景。

---

## 一句话总结（B1.5 核心）
- **退出码 0=成功**；`&&` 成功继续、`||` 失败兜底、`;` 无条件顺序。
- **管道默认只看最后一段**，中间失败会被掩盖 → 处理链条时 `set -o pipefail`，细查用 `PIPESTATUS`（且要立刻读）。
- **`()` 子shell 隔离**（变量/cd 不外泄），**`{}` 当前 shell**（会影响、共用重定向）。
- `A && B || C` **不是** if/else，逻辑重要时老老实实写 `if`。

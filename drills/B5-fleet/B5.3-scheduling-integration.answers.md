# KP B5.3 答案 — Scheduling Integration (cron + systemd timer)

> 脚本侧（日志包装、flock）和 `OnCalendar`/`.timer` 语法都真机验证过（`systemd-analyze calendar` 解析通过、`list-timers` 可见）。心法：**定时任务三件套——绝对路径、输出进日志、加锁防重叠。**

---

**1.** cron 五字段。
```
┌ 分(0-59)
│ ┌ 时(0-23)
│ │ ┌ 日(1-31)
│ │ │ ┌ 月(1-12)
│ │ │ │ ┌ 周(0-7，0/7=周日)
│ │ │ │ │
30 2 * * *   → 每天 02:30
*/15 * * * * → 每 15 分钟
0 */6 * * *  → 每 6 小时
0 9 * * 1    → 每周一 09:00
```
*为什么：* 从左到右"分时日月周"。`*`=任意、`*/N`=每 N、`a,b`=列举、`a-b`=范围。记不住可用 crontab.guru 之类核对。

**2.** crontab 条目。
```bash
crontab -e      # 编辑当前用户的 crontab
# 加一行：
30 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1
```
*为什么：* 每天 02:30 跑备份，输出追加进日志。`crontab -l` 看、`crontab -r` 清空（小心）。

**3.** PATH 坑（定时任务头号杀手）。
```bash
# cron 的 PATH 极简（通常只有 /usr/bin:/bin），你 shell 里能跑的命令它可能找不到
# 修法二选一：
# (a) 脚本里全用绝对路径： /usr/bin/aws  /usr/local/bin/foo
# (b) 脚本开头显式设 PATH：
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```
*为什么：* **"手动跑好好的，cron 里就失败"十有八九是 PATH**。cron 不加载你的 `.bashrc`，环境很干净。要么绝对路径，要么脚本头设 PATH。

**4.** 捕获输出。
```bash
... >> /var/log/job.log 2>&1
```
*为什么：* cron 默认把任务的输出**邮件**给用户（常常没配邮件=直接丢失）。`>> log 2>&1` 把 stdout+stderr 都存进日志，出问题能回溯。**定时任务不留日志=瞎跑。**

**5.** systemd `.service` + `.timer`（RHEL 10 主推）。
```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Nightly backup job

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh /data /backups 7
```
```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Run nightly backup at 02:30

[Timer]
OnCalendar=*-*-* 02:30:00
Persistent=true

[Install]
WantedBy=timers.target
```
*为什么：* 一个 `.service`（`Type=oneshot` 干一次就退）+ 同名 `.timer`（何时触发）。比 cron 强在：**日志自动进 journald**（`journalctl -u backup.service`）、`Persistent=true` 让**关机错过的任务开机补跑**、依赖关系可控。

**6.** 校验 OnCalendar。
```bash
systemd-analyze calendar '*-*-* 02:30:00'
#   Next elapse: Sun 2026-07-19 02:30:00 ...
```
*为什么：* 写完时间表达式**先用 `systemd-analyze calendar` 验一下下次触发时间对不对**，别等它半夜不跑才发现写错。常用简写：`daily`、`weekly`、`Mon *-*-* 09:00:00`。

**7.** 启用 + 查看。
```bash
systemctl daemon-reload
systemctl enable --now backup.timer     # 开机自启 + 立即激活
systemctl list-timers                    # 看下次触发时间
journalctl -u backup.service             # 看任务日志
```
*为什么：* 注意 **enable 的是 `.timer`**（不是 .service）。`list-timers` 一栏 NEXT/LEFT 看下次何时跑。

**8.** 安全的定时脚本（flock + 时间戳日志）。
```bash
#!/usr/bin/env bash
set -euo pipefail
exec 9>/var/run/backup.lock
flock -n 9 || { echo "already running, skip this tick"; exit 0; }   # 上轮没跑完就跳过
exec >>/var/log/backup.log 2>&1                                     # 之后全部进日志
echo "[$(date '+%F %T')] backup started"
# ... 真正工作 ...
echo "[$(date '+%F %T')] backup finished (rc=0)"
```
*为什么：* **`flock` 防重叠**——任务跑得比周期慢时，别让两个实例同时跑（B4.6/B3.7）；抢不到锁就干净退出(exit 0)。`exec >>log 2>&1` 让整段输出进日志、每行带时间戳。这就是"能安心挂进 cron/timer"的脚本长相。

---

## 一句话总结（B5.3 核心）
- **cron**：`分 时 日 月 周` + `>> log 2>&1`；当心 **PATH**（用绝对路径或脚本头设 PATH）。
- **systemd timer**（RHEL10 推荐）：`.service`(oneshot) + `.timer`(`OnCalendar` + `Persistent=true`)，`enable --now 的是 .timer`，日志进 journald。
- 写完 `systemd-analyze calendar` 验时间；`list-timers` 看下次。
- **脚本要 flock 防重叠 + 时间戳日志**，才能安心自动跑。

# KP B6.1 答案 — Debugging

> 全部真机跑过、输出核对无误。心法：**看不懂就 `bash -x` 跑一遍**，展开后的每一步都在眼前，变量拼错、引号漏了立刻现形。

---

**1.** `bash -x` 全脚本跟踪。
```bash
bash -x d.sh
# + x=5
# + y=10          ← $((x*2)) 已算成 10
# + echo 10
# 10
```
*为什么：* `-x` 把**每条命令展开后的实际形态**打到 stderr（前缀 `+`）。你能看到变量最终变成什么值——排障第一招。

**2.** `set -x` / `set +x` 局部跟踪。
```bash
set -x; echo hi; set +x; echo done
# + echo hi
# hi
# + set +x
# done
```
*为什么：* 只想看某段就 `set -x` 打开、`set +x` 关闭。大脚本里精确定位问题段，不被无关输出淹没。

**3.** 定制 `PS4` 显示行号。
```bash
export PS4='+${LINENO}: '     # 常用增强版：'+${BASH_SOURCE}:${LINENO}: '
set -x
```
*为什么：* 跟踪前缀默认是 `+`。把 `PS4` 设成含 `${LINENO}`（行号）、`${BASH_SOURCE}`（文件名）、`${FUNCNAME[0]}`（函数名），跟踪输出直接告诉你"哪个文件哪一行哪个函数"。多文件脚本排障神器。

**4.** `bash -n` 只查语法不执行。
```bash
bash -n bad.sh
# bad.sh: line 3: syntax error: unexpected end of file   （for 少了 done）
# 退出码 2；脚本一行都没真跑
```
*为什么：* `-n`（noexec）**只解析不运行**，上线前先扫一遍语法，避免"跑到一半才发现括号没闭合"。改完危险脚本，先 `bash -n` 再跑。

**5.** `set -v` 逐行回显源代码。
```bash
printf 'echo one\necho two\n' | bash -v
# echo one     ← 先回显这行源代码
# one          ← 再是它的输出
# echo two
# two
```
*为什么：* `-v`（verbose）把**读到的原始行**照打（不展开），配合 `-x`（展开后）一起用（`bash -xv`）能同时看到"写的是什么"和"跑成了什么"。

**6.** `declare -p` 查看变量真容。
```bash
arr=(a b c); declare -p arr
# declare -a arr=([0]="a" [1]="b" [2]="c")
```
*为什么：* `declare -p 变量` 打印它的**类型和确切内容**（含引号、下标）。怀疑数组元素带了空格、变量藏了空白字符时，比 `echo` 更靠谱。

**7.** `set -u` 抓拼写错误。
```bash
set -u; name="Tongzhou"; echo "$naem"
# bash: naem: unbound variable   （退出非 0）
```
*为什么：* 没 `-u` 时 `$naem`（拼错）当空串，脚本静默出错、极难查。`set -u` 让引用未定义变量直接报错——**变量名拼错这类 bug 的克星**（配合安全头 `set -euo pipefail`）。

**8.** 跟踪函数内部。
```bash
bash -x fn.sh
# + double 21
# + echo 42
# 42
```
*为什么：* `-x` 会跟进函数，展示函数内每步。想更清楚可在 `PS4` 里加 `${FUNCNAME[0]}` 标出当前函数名。

---

## 一句话总结（B6.1 核心）
- **`bash -x`**（或 `set -x`/`+x` 局部）= 看每条命令展开后的实际执行——最常用。
- **`PS4='+${BASH_SOURCE}:${LINENO}: '`** 让跟踪带文件名/行号。
- **`bash -n`** 上线前查语法（不执行）；**`set -v`** 回显源码行；**`bash -xv`** 双管齐下。
- **`declare -p`** 看变量真容；**`set -u`** 抓拼错的变量名。
